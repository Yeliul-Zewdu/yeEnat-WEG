# YeEnat Weg — MVP Technical Design

## Database Schema · API Design (MVP only)

> Scope: Hackathon MVP (demo June 14, 2026). Covers the end-to-end core loop:
> **register → health profile → AI weekly meal plan + lifestyle suggestion → log meals
> (follow plan or insert the ingredients you actually ate) → record daily blood sugar
> (diabetics) → plan auto-updates.**
> Anything outside this loop (community, B2B/corporate, gamification, notifications
> service, shopping lists, barcode/photo logging, weekly AI reports, offline sync) is
> **deferred** — see [§4 Deferred](#4-deferred-post-mvp).

---

## 0. Core Concept: Meal = Ingredients

A **meal** is what a person eats at a slot (breakfast / lunch / dinner / snack), and it is
**built from ingredients with quantities** — e.g. *"Oats 100 g + 5 eggs"*. `oats` and `eggs`
are **ingredients** with nutrition stored per 100 g (or per unit).

The same `meals` + `meal_items` structure is reused in two places:

1. **Planned meal** — generated by the AI and attached to a meal-plan slot.
2. **Logged meal** — what the user actually ate. If they followed the plan, the log points at
   the planned meal. If they deviated, the user **inserts the ingredients** of the meal they had,
   which creates a new logged meal.

```
ingredients ──< meal_items >── meals ──┬── meal_plan_slots   (planned)
                                       └── meal_logs          (actually eaten)
```

## Part 1 — PostgreSQL Schema (Server)

### 1.1 Users & Authentication

```sql
CREATE TABLE users (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    phone_number    VARCHAR(20) UNIQUE,
    google_uid      VARCHAR(128) UNIQUE,
    full_name       VARCHAR(120) NOT NULL,
    preferred_lang  VARCHAR(10) NOT NULL DEFAULT 'am',   -- 'am' | 'en'
    is_active       BOOLEAN NOT NULL DEFAULT TRUE,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    deleted_at      TIMESTAMPTZ                            -- soft delete (data erasure)
);

-- One-time passwords for phone auth
CREATE TABLE otp_sessions (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    phone       VARCHAR(20) NOT NULL,
    otp_hash    VARCHAR(128) NOT NULL,
    expires_at  TIMESTAMPTZ NOT NULL,
    used        BOOLEAN NOT NULL DEFAULT FALSE,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Long-lived refresh tokens (access tokens are stateless JWTs)
CREATE TABLE refresh_tokens (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id     UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    token_hash  VARCHAR(256) NOT NULL UNIQUE,
    expires_at  TIMESTAMPTZ NOT NULL,
    revoked_at  TIMESTAMPTZ,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### 1.2 Health Profile

```sql
-- One profile per user. Drives meal-plan + lifestyle generation.
CREATE TABLE health_profiles (
    id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id             UUID NOT NULL UNIQUE REFERENCES users(id) ON DELETE CASCADE,

    -- Physical metrics
    sex                 VARCHAR(10) NOT NULL,             -- 'male' | 'female' | 'other'
    birth_year          SMALLINT NOT NULL,
    height_cm           NUMERIC(5,1) NOT NULL,
    current_weight_kg   NUMERIC(5,1) NOT NULL,
    target_weight_kg    NUMERIC(5,1),
    bmi                 NUMERIC(4,1) GENERATED ALWAYS AS
                            (current_weight_kg / ((height_cm/100)*(height_cm/100))) STORED,

    -- Activity & goal
    activity_level      VARCHAR(20) NOT NULL DEFAULT 'sedentary',
                        -- 'sedentary' | 'light' | 'moderate' | 'active'
    primary_goal        VARCHAR(30) NOT NULL,
                        -- 'lose_weight' | 'gain_weight' | 'maintain' | 'manage_condition'

    -- Lifestyle inputs (used for the sleep/exercise suggestion)
    wake_time           TIME,
    sleep_time          TIME,

    -- Dietary preferences (folded in — no separate table for MVP)
    fasting_type        VARCHAR(20) NOT NULL DEFAULT 'none', -- 'none' | 'orthodox' | 'ramadan'
    is_vegetarian       BOOLEAN NOT NULL DEFAULT FALSE,
    is_vegan            BOOLEAN NOT NULL DEFAULT FALSE,
    allergies           TEXT[] NOT NULL DEFAULT '{}',

    -- Nutrition targets (computed on save: Harris–Benedict + goal/condition adjustment)
    daily_kcal_target   SMALLINT,
    protein_g_target    SMALLINT,
    carb_g_target       SMALLINT,
    fat_g_target        SMALLINT,
    sodium_mg_target    SMALLINT,                          -- tightened for hypertension
    sugar_g_target      SMALLINT,                          -- tightened for diabetes

    profile_version     INTEGER NOT NULL DEFAULT 1,        -- bumped on every edit
    created_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at          TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Lookup of supported conditions
CREATE TABLE medical_conditions (
    id      SMALLSERIAL PRIMARY KEY,
    code    VARCHAR(30) UNIQUE NOT NULL,   -- 'diabetes_t1' | 'diabetes_t2' | 'hypertension'
    label   VARCHAR(80) NOT NULL
);

-- User ⇄ condition (drives disease-aware planning + whether sugar tracking is required)
CREATE TABLE user_conditions (
    user_id         UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    condition_id    SMALLINT NOT NULL REFERENCES medical_conditions(id),
    diagnosed_at    DATE,
    PRIMARY KEY (user_id, condition_id)
);
```

### 1.3 Ingredients (Ethiopian Food Database)

```sql
-- An ingredient is a single food with nutrition per 100 g.
CREATE TABLE ingredients (
    id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name_am             VARCHAR(120) NOT NULL,    -- Amharic name
    name_en             VARCHAR(120) NOT NULL,    -- English name

    -- Nutrition per 100 g
    kcal                NUMERIC(6,1) NOT NULL,
    protein_g           NUMERIC(5,2) NOT NULL,
    carb_g              NUMERIC(5,2) NOT NULL,
    fat_g               NUMERIC(5,2) NOT NULL,
    fiber_g             NUMERIC(5,2),
    sodium_mg           NUMERIC(6,1),
    sugar_g             NUMERIC(5,2),
    glycemic_index      SMALLINT,                 -- used for diabetic carb control

    -- Flags used by the planner
    is_fasting_safe     BOOLEAN NOT NULL DEFAULT FALSE,   -- Orthodox fasting
    is_vegetarian       BOOLEAN NOT NULL DEFAULT FALSE,
    is_vegan            BOOLEAN NOT NULL DEFAULT FALSE,

    -- Household-measure helper (e.g. "1 medium injera (200g)", "1 egg (50g)")
    default_serving_g   NUMERIC(6,1) NOT NULL DEFAULT 100,
    serving_description VARCHAR(100),

    is_active           BOOLEAN NOT NULL DEFAULT TRUE,
    created_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at          TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### 1.4 Meals (composition of ingredients)

```sql
-- A meal = a named set of ingredients. Created by the AI (planned) or by the user
-- (logged substitution). Nutrition totals are aggregated from meal_items on write.
CREATE TABLE meals (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name_am         VARCHAR(150),
    name_en         VARCHAR(150),
    meal_type       VARCHAR(20) NOT NULL,     -- 'breakfast' | 'lunch' | 'dinner' | 'snack'
    source          VARCHAR(20) NOT NULL,     -- 'ai_plan' | 'user_logged'
    created_by      UUID REFERENCES users(id) ON DELETE SET NULL,  -- NULL for AI meals

    -- Aggregated nutrition (sum over meal_items, for fast display)
    total_kcal      NUMERIC(7,1) NOT NULL DEFAULT 0,
    total_protein_g NUMERIC(6,2) NOT NULL DEFAULT 0,
    total_carb_g    NUMERIC(6,2) NOT NULL DEFAULT 0,
    total_fat_g     NUMERIC(6,2) NOT NULL DEFAULT 0,
    total_fiber_g   NUMERIC(6,2),
    total_sodium_mg NUMERIC(7,1),
    total_sugar_g   NUMERIC(6,2),
    glycemic_load   NUMERIC(6,2),

    is_fasting_safe BOOLEAN NOT NULL DEFAULT FALSE,   -- derived from ingredients
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- The ingredients that make up a meal, with quantity (e.g. oats 100 g, eggs 250 g)
CREATE TABLE meal_items (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    meal_id         UUID NOT NULL REFERENCES meals(id) ON DELETE CASCADE,
    ingredient_id   UUID NOT NULL REFERENCES ingredients(id),
    quantity_g      NUMERIC(7,2) NOT NULL,
    UNIQUE (meal_id, ingredient_id)
);
```

### 1.5 Meal Plan (weekly)

```sql
-- A generated weekly plan. A new row replaces the old one when regenerated
-- (weekly refresh, profile change, or blood-sugar-driven update).
CREATE TABLE meal_plans (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    week_start      DATE NOT NULL,
    status          VARCHAR(20) NOT NULL DEFAULT 'active',  -- 'active' | 'replaced'
    profile_version INTEGER NOT NULL,                       -- snapshot used to generate
    trigger_reason  VARCHAR(30) NOT NULL DEFAULT 'initial',
                    -- 'initial' | 'weekly_refresh' | 'profile_change' | 'blood_sugar'
    generated_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Exactly one active plan per user
CREATE UNIQUE INDEX idx_one_active_plan
    ON meal_plans (user_id) WHERE status = 'active';

CREATE TABLE meal_plan_days (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    plan_id     UUID NOT NULL REFERENCES meal_plans(id) ON DELETE CASCADE,
    plan_date   DATE NOT NULL,
    day_kcal_target NUMERIC(7,1),
    UNIQUE (plan_id, plan_date)
);

-- One slot per meal. Points at the planned meal (composition of ingredients).
CREATE TABLE meal_plan_slots (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    day_id      UUID NOT NULL REFERENCES meal_plan_days(id) ON DELETE CASCADE,
    slot_type   VARCHAR(20) NOT NULL,          -- 'breakfast' | 'lunch' | 'dinner' | 'snack'
    slot_index  SMALLINT NOT NULL DEFAULT 0,   -- for a 2nd snack
    meal_id     UUID NOT NULL REFERENCES meals(id),
    is_logged   BOOLEAN NOT NULL DEFAULT FALSE,
    UNIQUE (day_id, slot_type, slot_index)
);
```

### 1.6 Logging — Meals, Weight, Blood Sugar

```sql
-- Daily header (one per user per day). Groups meal logs + water.
CREATE TABLE daily_logs (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id     UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    log_date    DATE NOT NULL,
    water_ml    SMALLINT NOT NULL DEFAULT 0,
    UNIQUE (user_id, log_date)
);

-- What the user actually ate for a slot.
--   adherence = 'followed'    → meal_id is the planned meal (plan_slot_id set)
--   adherence = 'substituted' → meal_id is a new 'user_logged' meal built from the
--                               ingredients the user inserted
CREATE TABLE meal_logs (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    daily_log_id    UUID NOT NULL REFERENCES daily_logs(id) ON DELETE CASCADE,
    slot_type       VARCHAR(20) NOT NULL,      -- 'breakfast' | 'lunch' | 'dinner' | 'snack'
    plan_slot_id    UUID REFERENCES meal_plan_slots(id) ON DELETE SET NULL,
    meal_id         UUID NOT NULL REFERENCES meals(id),
    adherence       VARCHAR(20) NOT NULL,      -- 'followed' | 'substituted'
    logged_at       TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE weight_logs (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id     UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    weight_kg   NUMERIC(5,1) NOT NULL,
    logged_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Daily self-reported health readings that drive plan updates:
--   • blood_sugar    (diabetes)     → value_mg_dl
--   • blood_pressure (hypertension) → systolic_mm_hg / diastolic_mm_hg
-- An out-of-range reading regenerates the plan (meal_plans.trigger_reason).
CREATE TABLE health_readings (
    id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id          UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    reading_type     VARCHAR(20) NOT NULL,          -- 'blood_sugar' | 'blood_pressure'

    value_mg_dl      NUMERIC(6,1),                  -- blood_sugar only
    systolic_mm_hg   SMALLINT,                      -- blood_pressure only
    diastolic_mm_hg  SMALLINT,                      -- blood_pressure only

    context          VARCHAR(20) NOT NULL DEFAULT 'random',
                     -- sugar: 'fasting' | 'pre_meal' | 'post_meal' | 'random'
    measured_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    note             TEXT,
    CONSTRAINT chk_reading_value CHECK (
        (reading_type = 'blood_sugar'    AND value_mg_dl IS NOT NULL) OR
        (reading_type = 'blood_pressure' AND systolic_mm_hg IS NOT NULL
                                         AND diastolic_mm_hg IS NOT NULL)
    )
);
```

### 1.7 Lifestyle Suggestion

```sql
-- Generated alongside each meal plan: a sleep routine + a few exercise suggestions.
-- MVP displays these (the SRS "lifestyle module"); heavy sleep/exercise logging is deferred.
CREATE TABLE lifestyle_plans (
    id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id             UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    meal_plan_id        UUID REFERENCES meal_plans(id) ON DELETE CASCADE,
    status              VARCHAR(20) NOT NULL DEFAULT 'active',  -- 'active' | 'replaced'

    sleep_target_start  TIME,
    sleep_target_end    TIME,
    sleep_note_am       TEXT,
    sleep_note_en       TEXT,

    -- [{ "name_en": "Brisk walk", "category": "low_impact", "duration_min": 30,
    --    "days": ["mon","wed","fri"] }, ...]
    exercise_suggestions JSONB NOT NULL DEFAULT '[]',

    generated_at        TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE UNIQUE INDEX idx_one_active_lifestyle
    ON lifestyle_plans (user_id) WHERE status = 'active';
```

### 1.8 Indexes

```sql
CREATE INDEX idx_ingredients_fasting   ON ingredients (is_fasting_safe, is_active);
CREATE INDEX idx_ingredients_fts       ON ingredients
    USING gin (to_tsvector('simple', name_am || ' ' || name_en));
CREATE INDEX idx_meal_items_meal       ON meal_items (meal_id);
CREATE INDEX idx_plan_days_plan        ON meal_plan_days (plan_id, plan_date);
CREATE INDEX idx_plan_slots_day        ON meal_plan_slots (day_id, slot_type);
CREATE INDEX idx_meal_logs_daily       ON meal_logs (daily_log_id, slot_type);
CREATE INDEX idx_weight_logs_user      ON weight_logs (user_id, logged_at DESC);
CREATE INDEX idx_health_readings_user  ON health_readings (user_id, reading_type, measured_at DESC);
```

## Part 2 — API Design (REST · FastAPI)

> **Base URL** `https://api.yeenatweg.com/v1`
> **Auth** Bearer JWT (15 min access) + refresh token
> **Content-Type** `application/json` · **Localization** `Accept-Language: am | en`

### 2.1 Authentication

| Method | Path | Description |
|---|---|---|
| `POST` | `/auth/otp/send` | Send OTP to a phone number |
| `POST` | `/auth/otp/verify` | Verify OTP → returns tokens |
| `POST` | `/auth/google` | Google Sign-In (`id_token` in body) |
| `POST` | `/auth/refresh` | Rotate access token |
| `POST` | `/auth/logout` | Revoke refresh token |

**`POST /auth/otp/verify` → 200**
```json
{
  "access_token": "eyJ...",
  "refresh_token": "eyJ...",
  "expires_in": 900,
  "user": { "id": "uuid", "full_name": "Tigist Bekele", "is_new": true }
}
```
`is_new: true` → client routes to onboarding (health profile).

### 2.6 Logging (Meals · Water · Weight)

| Method | Path | Description |
|---|---|---|
| `GET` | `/logs/daily/{date}` | Daily totals vs. targets + entries |
| `POST` | `/logs/daily/{date}/meals` | Log a meal (followed **or** substituted) |
| `DELETE` | `/logs/meals/{meal_log_id}` | Delete a meal log |
| `POST` | `/logs/daily/{date}/water` | Add water (`{ "amount_ml": 250 }`) |
| `POST` | `/logs/weight` | Log body weight |
| `GET` | `/logs/weight?days=30` | Weight trend |

**A) Followed the plan** — `POST /logs/daily/2026-06-08/meals`
```json
{ "plan_slot_id": "uuid", "adherence": "followed" }
```

**B) Ate something different** — user inserts the ingredients of what they actually had:
```json
{
  "plan_slot_id": "uuid",
  "adherence": "substituted",
  "slot_type": "breakfast",
  "meal": {
    "name_en": "Firfir & tea",
    "items": [
      { "ingredient_id": "uuid", "quantity_g": 200 },
      { "ingredient_id": "uuid", "quantity_g": 150 }
    ]
  }
}
```
The server creates a `user_logged` meal from the items, aggregates its nutrition, links the
`meal_log`, and marks the slot `is_logged = true`. **→ 201** returns the created meal log with
computed nutrition.

**`GET /logs/daily/{date}` → 200**
```json
{
  "date": "2026-06-08",
  "targets":  { "kcal": 1800, "protein_g": 95, "carb_g": 190, "sodium_mg": 2000, "sugar_g": 30 },
  "consumed": { "kcal": 1240, "protein_g": 62, "carb_g": 155, "sodium_mg": 1100, "sugar_g": 22 },
  "water_ml": 1500,
  "entries": [
    { "meal_log_id": "uuid", "slot_type": "breakfast",
      "meal_name_en": "Firfir & tea", "adherence": "substituted", "kcal": 420 }
  ],
  "sodium_alert": false,
  "sugar_alert": false
}
```

### 2.7 Blood Sugar (Diabetes) → triggers plan update

| Method | Path | Description |
|---|---|---|
| `POST` | `/health-readings` | Record a blood-sugar reading |
| `GET` | `/health-readings?type=blood_sugar&days=30` | Reading history |

**`POST /health-readings` — request**
```json
{ "reading_type": "blood_sugar", "value_mg_dl": 184, "context": "fasting" }
```
**→ 201** — if the reading is out of the healthy range, the meal plan is regenerated with
tightened carb/sugar targets and the response reports it:
```json
{
  "reading_id": "uuid",
  "value_mg_dl": 184,
  "status": "high",
  "plan_updated": true,
  "new_plan_id": "uuid",
  "message_en": "Your fasting sugar is high. Your plan was adjusted to lower-glycemic meals."
}
```
*(When the reading is in range, `plan_updated` is `false`.)*

## 3. End-to-End MVP Flow

```
1. Register            POST /auth/otp/verify              → tokens, is_new=true
2. Health profile      PUT  /users/me/health-profile      → targets + plan + lifestyle generated
3. View plan           GET  /meal-plans/current           → 7 days of meals (ingredients) + lifestyle
4. Log a meal
     • followed         POST /logs/daily/{date}/meals  {adherence:"followed", plan_slot_id}
     • substituted      POST /logs/daily/{date}/meals  {adherence:"substituted", meal:{items:[…]}}
5. See dashboard       GET  /logs/daily/{date}            → consumed vs. target, alerts
6. (Diabetic) sugar    POST /health-readings  {value_mg_dl} → plan auto-updates if out of range
7. Repeat weekly       POST /meal-plans/generate (or auto weekly_refresh)
```
